PPP协议

引言
...

对于点对点数据链路的控制 一个发送端、一个接收端、一条链路

  • 无需介质访问控制
  • 无需明确的 MAC 寻址

常见的点对点数据链路控制协议：

  • HDLC
  • PPP

功能
...

  • 组帧：将网络层数据报封装到数据链路层帧中
    • 可以承载任何网络层协议分组
    • 可以向上实现分用
  • 比特透明传输
    • 数据与必须支持承载任何比特模式
  • 差错检测（无纠正）
  • 连接活性检测
  • 网络层地址协商：端结点可以学习/配置彼此网络地址

无需支持的功能
...

  • 差错控制纠正/恢复
  • 流量控制
  • 乱序交付控制
  • 多点链路

帧结构
...

| 名称 | 意义 | 备注 | | -------- | ------ | -------------------------------- | | flag | 定界符 | 表示帧的开始和结束 | | Address | 地址 | 无意义，全 1 | | Control | 控制 | 无意义 | | Protocol | 协议 | 指示了上层协议，通过协商确定长度 | | info | 信息 | 上层协议分组数据 | | check | 校验 | 采用 CRC 校验，用于差错控制 | 双方可以协商是否包含地址字段和控制字段，并且控制协议与校验字段的长度

透明比特传输实现
...

透明传输是指不管所传的数据是什么样的比特组合，都应当能够在链路上传送

由于在 info 中，可能会出现定界符的信息<01111110>，会导致接收方误以为数据帧结束

所以发送端需要在数据发送前进行处理： 如果数据中存在<01111110>和<01111101>字节，那么就在字节前额外添加<01111101>

接收端收到信息时，如果有单个<01111101>, 那么就表示一个填充字节，如果连续两个<01111101>，就丢弃第一个，将第二个作为数据接收 如果有<01111110>跟在<01111101>后面，就说明这个<01111110>不是一个定界符，作为数据接收

协商
...

在交换网络层数据之前，PPP 数据链路两端必须：

  • 配置 PPP 链路
    • 最大帧长
    • 身份认证
    • 帧信息长度控制
  • 学习/配置网络层信息（对于 IP 协议）
    • 通过交换 IPCP 协议报文，完成 IP 地址等相关信息配置
