特点
...

  • 点对点：客户机/服务器间进程需要建立连接
  • 可靠的传输

序列号

  • 序列号指的是 segment 中的第一个字节的编号
  • 一个字节占用一个序列号，n 个字节的 segment 占用 n 个序列号，发送时只需发送第一个字节的序列号
  • 建立 TCP 连接时，双方随即选择序列号 ACKs：
  • 希望接收到的下一个字节的序列号
  • 累计确认：该序列号之前的所有字节已被确认收到 对于乱序到达的 segment：
  • 规范中无规定，由 TCP 实现者决策

TCP 超时时间
...

SampleRTT：测量从段发出去收到 ACK 的时间

  • 忽略重传 测量多个 SampleRTT，求加权平均值，形成 RTT 的估计值
    一般的
    这样参考了以前的 RTT，避免因为偶尔的波动而对 RTT 影响过大

超时器在估计 RTT 的基础上，还需要加上一部分时间 超时器设置为：EstimatedRTT+“安全边界”

若 RTT 波动过大，则安全边界需要更大

  • 如果收到同一个 Segment 的三个 ACK
    • 快速重传

  • 等待 500ms 看是否有下个段
  • 如没有，发送 ACK

收到所期望序列号的段，且之前有段等待确认
...

  • 发送 ACK 确认

收到乱序段
...

  • 发送重复 ACK，指示期望需要的段的序列号

快速重传机制
...

TCP 的实现中，如果发生超时，超时时间间隔将重新设置，即将超时时间间隔加倍，导致其很大

重发丢失的分组之前，要等待很长时间

但是可以通过重复 ACK 检测分组丢失

如果 Sender 收到对同一个数据的三个冗余 ACK，则进行快速重传：忽略超时器，立刻重传该 ACK 段

流量控制机制
...

通过 RcvWindow 信息，接收方告知发送方自己当前缓存的可用空间

发送方限制自己已经发送但还未收到 ACK 的数据不超过接收方的空闲 RcvWindow

当接收方告知发送方 RcvWindow=0，发送方仍然可以发送一个很小的数据 防止死锁

发送方可以通过调整自己的窗口大小，来进行流量控制

...

TCP sender 和 rceiver 在传输数据前需要建立连接

初始化 TCP 变量
...

  • seq
  • Buffer 和流量控制信息

客户端-连接发起者 服务器-等待客户端请求

三次握手（连接建立）
...

  • 客户端发送 SYN=1 段
    • 客户端选择初始序列号
    • 客户端无数据
  • 服务器发送 SYN=1 段的 ACK
    • 服务器初始化缓存
    • 服务器确认初始序列号
  • 客户端发送 ACK

连接关闭
...

双方都可以发起关闭

  • 客户端发送 FIN=1 段
  • 服务器回复 ACK 关闭连接，发送 FIN=1 段
  • 客户端回复 ACK，进入等待 30s：如果收到 FIN=1 段，重新发送 ACK

拥塞控制
...

将 CongWin 设置为窗口的动态大小 通过调整窗口的大小，来控制同一时间发送出去但没有接收到的数据量，以达到控制网络速度的效果 用来进行拥塞控制 当发生 loss 事件=超时或三个冗余 ACK 发送方降低速率

AIMD 加性增，乘性减
...

逐渐增加发送速率，谨慎探测可用带宽，直到发生 loss

当发生 loss，应当快速降速

所以需要慢慢加速，很快速的减速

MSS：最大报文段长度

  • 每个 RTT 将 CongWin 增大一个 MSS——拥塞避免
  • 发生 loss 后 CongWin 减半

SS 慢启动
...

连接开始时，指数性增长 初始化

CongWin = 1
for (each segment ACKed)
	CongWin++
until (loss event OR CongWin>threshold)

Threshold: Loss 事件发生时，被设置为 Loss 事件前 CongWin 的 1/2

Loss 事件处理
...

三个冗余 ACKs
...

  • CongWin 切到一半
  • 开始线性增长

超时
...

  • CongWin 设为 1 个 MSS
  • 然后指数增长
  • 达到 threshold 后再线性增长

原因
...

3 个冗余 ACKs 表示网络还能传输一些段 超时表示拥塞事件更为严重，所以降为 1 个 MSS
