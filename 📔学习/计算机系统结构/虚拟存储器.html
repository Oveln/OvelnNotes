简介
...

通过辅助映射表建立虚、实地址的映射。虚地址空间可以远大于主存空间（一部分是数据存在主存，另一部分存在辅存）
当程序员设计的程序远远大于实际地址空间时，将程序空间分割成较小的段和页，系统程序按需要调入相应的物理主存，通过辅助映射表管理
辅助映射表的相关操作不需要程序员设置，这些对程序员都是透明的

如果访问的数据不在 中，该程序暂停执行或被挂起，直到数据被存放到 中
只能服务于单用户环境

管理方式
...

虚拟存储器通过增设辅助映射表来构建程序在主存中的定位
通过将程序分成若干个段和页，用相应的映射表指明该程序某段或某页是否装入主存

  • 如果装入，指明其在主存中的起始地址
  • 如未装入，去辅存中调段或页，装入主存候在映像表中建立映射关系

段式管理
...

将大型程序分解成多个在逻辑上相对独立的模块，每个模块都是一个单独的段。以起点为 0 相对编址
当某个段由辅存调入主存时，系统赋予该段一个基址（该段存放在主存中的起始地址）
通过基址和单元在段内的相对位移来形成单元在主存中的实际地址

这种管理方式称为段式管理

具体实现
...

每道程序在系统中会维护一个段表，记录该程序每段的段号、装入主存中的起始地址、是否装入主存、段长、访问方式
同时，在段表基址寄存器中会维护一个表，记录程序号、段号、段内位移

  • CPU 寻虚地址，检索段表
  • 通过程序号在段表基址寄存器中找到对应的段表基地址
  • 通过段号找到程序的段表，检索段表
    • 对应段已装入主存
      • 通过段表信息找到对应的地址
      • 通过起始地址+段内位移，获得主存中的地址
    • 对应段未装入主存
      • 将对应段装入主存，维护段表信息和段表基址寄存器信息

访问方式可以记录该段是否只读、是否只执行
同时的，段表本身也是一个段

特点
...

  • 各段按逻辑特点组合，容易以段为单位进行存储保护
    • 有的段只读
    • 有的段不能作为指令执行
    • 有的段只能执行
    • 一旦违反就中断
  • 段的管理由于是逻辑分解的，长度是不定的，且段在主存中的起点是不随意的
  • 还需要建立一个实主存管理表，指明主存中哪些位置被占用，哪些没有
    • 当段从辅存装入主存时，就在占用区域表中增加一项，修改可用区域表
    • 当段从主存移出时，在占用表中对应的项，移入可用区域表，并进行有关的可能的归并操作
  • 当某个程序结束或被优先级更高的程序所取代，也移出可用区域表，修改占用区域表

页式管理
...

将主存空间和程序空间机械性地等分成固定大小的页（512B 到几 KB 不等）
按顺序编号，通过映射表管理虚页和实页的对应

在页式存储中，一个程序的起点必然处于一个页的起点，程序中每个虚地址就由虚页号和页内位移字段组成

具体实现
...

每道程序会维护一个页表，记录主存起点、装入位、访问方式、专用位
同时主存也被分为了诸多登场的页
页表基址寄存器维护第 i 行（虚页号）对应的实页号

  • CPU 寻虚地址
    • 对应页已装入
      • 在页表中找到对应的实页号
      • 通过用户表示号（进程号）在页表基址寄存器中找到对应的程序页表起点
      • 通过实页号和页表起点找到对应的实页基址
      • 实页基址+页内偏移获得实际地址
    • 对应页未装入
      • 将对应虚页装入主存，维护页表，维护基址寄存器

总结
...

  • 段式管理需要反复将起始地址加上段内位移的操作，页式管理只需要将主存实页号和页内位移组装起来，就可以获得主存地址
  • 页表不需要有页号，因为页号和表行号是对应的，也不需要页长字段

段页式管理
...

段式管理和页式管理各有优缺点

  • 页式
    • 程序员完全透明，所需映射表硬件少，地址变换快
    • 页内零头不能完全利用，会有浪费，但浪费的仍然比段式少
  • 段式
    • 每个段独立，利于程序员实现链接
    • 段的扩大、缩小、修改不会影响其他段
    • 每个段只包含一个类型的对象

为了取长补短，引入了段页式的管理

  • 将主存分成固定大小的页，程序按模块分段、每个段分成和实主存相同的页

...

每个程序维护一个段表和一个页表
段表信息包括，是否装入、页表起点、访问方式、段长
CPU 维护一个段表基址寄存器

  • CPU 访问虚地址
    • 通过用户标识获得段表基址
    • 找到段表，通过段表信息获得页表起点
    • 通过虚页号和页表起点获得实页号
    • 通过实页号和页内偏移获得实际地址
  • 如果访问页表时，装入位为 0，就装入该页，并维护段表和页表

所以存在两个虚页同时映射到同一个实页上的情况，映像方式应该保证尽量减少页面争用的情况

由于 使得页表中绝大部分行中的页号 字段和其他字段都无效了，会降低效率

解决办法一
...

将页表中装入位为 0 的行用实页号 字段存放该程序此虚页在辅存中的实地址，以便调页的时候实现用户虚页号到辅存实地址的变换
不过当辅存虚地址的位数与用户虚页号字段位数差别大时，就很难利用

解决办法二（目录表法）
...


将页表压缩成只存放已装入主存的那些虚页和实页的对应关系

  • 用基号 b 和 对应
  • 该表最多 行
  • 该表采用按内容访问的相联存储器
    - 通过基号+用户虚页号的组合在相联存储器中查找实页号
    - 如果找到：
    - 返回实页号，和页内位移组合成实际地址
    - 如果没有找到：
    - 发出页面失效故障，从辅存中调页
    目录表法不需要设置装入位

外页表
...

当发生页面失效时，需要从辅存调入数据，那么就需要虚页和辅存地址的映射关系
将这个映射关系作为外页表维护
将用于内部地址变换的表称作内页表，将用于外部地址变换的表称作外页表
由于虚拟存储器的页面失效率一般低于 1%，调用外页表进行虚地址和辅存实地址的变换机会很少，加上辅存访问调页速度慢，因此外页表通常存放在辅存中
只有程序初始运行，才会把外页表的内容转录到已建立的内页表的实页号地址字段中

外页表的地址变换一般由软件实现，因为辅存访问速度慢，就算用硬件实现，CPU 也是在空转

页面替换算法
...

当处理机要用的指令或数据不在主存中时，会产生页面失效，需要从辅存中调入数据到主存

通常虚存空间比主存空间大的多，必然会出现主存已满又发生页面失效的情况，需要选择主存中的哪个页来进行替换，这就是替换算法解决的问题。

替换算法的确定主要看主存是否有高的命中率，也要看算法是否便于实现，成本是否低

随机算法RAND
...

采用软硬件的随机数生成器来产生要被替换的页号，简单便于实现，但没有利用主存使用的历史信息，反应不了程序的局部性，主存的命中率低，现在已经不采用了

先进先出算法FIFO
...

最早被装入主存的页作为被替换的页，这种算法实现简单方便，只要为每个页配一个计数器字段即可，每当一页装入主存，就让该页的计数器清零，其他已经装入主存的页的计数器+1
需要替换时，就选择计数器最大的页
这种算法虽然利用了主存的历史信息，但不一定能反映出程序的局部性，因为最先进入的页很可能时现在经常使用的页

近期最少使用算法（主存页面表）LRU
...

选择近期最少访问的页作为被替换的页，这种算法能比较正确的反应程序的局部性

但完全按照此算法实现比较困难，所以一般使用其变形，即把最近最久未被访问的页作为被替换页

以上为操作系统为实现主存管理设置的主存页面表，每一行用来记录主存中各页的使用状况
主存页面表有且只有一个，在主存中进行维护
占用位：标识是否被使用
使用位：

  • 开始时，所有的使用位为 0
  • 如果某个单元访问过该页，硬件自动将该位置 1
  • 只有当所有的占用位为 1，且要调入页时，会发生页面失效，需要页面替换，使用使用位为 0 的页替换

使用位维护：

  • 使用位不能出现全 1，否则不能确定哪页被替换
  • 随机期法：
    • 一种办法是一旦使用位要变为全 1 时，立即由硬件将所有使用位归 0
  • 定期加 ：
    • 定期扫描所有使用位
    • 若使用位为 0，将期 +1，使用位仍为 0
    • 若使用位为 1，则将 =0，使用位归零
    • 这样以来，最大的页就是最久未使用的页，将被替换

这种方法比近期最少使用法所消耗的计数器硬件少得多，且由于页面失效后调页时间长，加上程序换道，因此主存页面表的修改可以软硬件结合实现

主存页面表可以增设修改位，如果需要写回辅存时，该页没有被修改过，则可以不用写回

优化替换算法OPT
...

将未来近期不需要的页替换出去，这种算法称为优化替换算法
但需要知道这个信息，得让程序运行过一次，这是不现实的
这是一种理想算法，可以用来作为评价其他算法好坏的标准
